µProcessador 6 — "Somador de Loop"

Explicação completa do LAB6: extensão de sinal, constantes de 5 bits, flush de branches condicionais e os 3 bugs corrigidos.

1 · Visão Geral

O LAB6 estende o processador do LAB5 com desvios condicionais (BLE e BCC). O programa de teste calcula a soma 0 + 1 + 2 + … + 29 = 435 usando um loop controlado por BLE (condição R3 < 30), e copia o resultado para R5 ao final.

Resultado esperado ao término da simulação:

R3 = 30 (0x001E) — contador, valor no qual R3<30 deixa de ser verdade
R4 = 435 (0x01B3) — acumulador da soma 0+1+2+…+29
R5 = 435 (0x01B3) — cópia de R4 após o loop

Ao longo do desenvolvimento foram encontrados e corrigidos 3 bugs que impediam esses resultados. Cada bug é explicado em detalhes na seção 6.

2 · O que cada arquivo faz

top-level

processador6.vhd

Nível mais alto. Instancia e conecta todos os componentes. Contém o mux de flush e, novidade do LAB6, a lógica de extensão de sinal do campo cte5 do IR (5 bits → 16 bits). Essa extensão alimenta tanto o SUBI quanto o LD.

ROM

rom.vhd

Memória de programa síncrona, 128 posições × 15 bits. Captura o endereço na borda de subida do clock e apresenta o dado no ciclo seguinte (latência de 1 ciclo). Essa latência é a origem dos problemas de flush também em branches condicionais.

Registrador

pc.vhd

Contador de programa de 7 bits. Atualizado com PC+1 no estado 01, e com o endereço de destino do desvio no estado 10 (quando JMP, BLE ou BCC executam).

Registrador

ir.vhd

Instruction Register de 15 bits. O wr_en é ativado somente no estado 00 (fetch). É exatamente nesse ciclo que o flush deve agir — forçando zero (NOP) na entrada do IR quando há desvio.

Controle

uc6.vhd

Unidade de controle. Novidades vs LAB5: decodifica os opcodes BLE (0101) e BCC (0110); aceita entradas flag_ble_i e flag_bcc_i; gera addr_abs_s como destino absoluto dos branches. O flush e o ajuste de PC (target−1) seguem a mesma lógica do JMP.

Controle

maq_estados.vhd

Contador síncrono de 2 bits: 00 → 01 → 10 → 00 → … (fetch → decode → execute). Idêntico ao LAB5. Cada instrução consume exatamente 3 ciclos de clock.

Dados

banco_regs.vhd

Banco de 8 registradores de 16 bits (R0–R7). Decodificador 1-de-8 para escrita; dois MUXes de leitura independentes. Os sinais internos q_r0…q_r7 são visíveis no GTKWave.

Dados

ULA.vhd

ULA de 16 bits, operações ADD e SUB. Calcula flags C, Z, N, V e BLE e BCC:
• BLE = 1 quando Z=1 ou N≠V (resultado ≤ 0 com sinal)
• BCC = 1 quando C=0 (sem borrow / sem carry)

Novo no LAB6

reg1bit.vhd (flags)

Flip-flop de 1 bit com wr_en. Dois são instanciados no top-level para armazenar flag_ble e flag_bcc. São atualizados somente quando instruções aritméticas executam (SUBI ou ADD). O BLE é testado pelo SUBI antes do branch.

Testbench

processador_tb.vhd

Gera clock de 100 ns, reset de 200 ns, simula 100 µs. A verificação é feita visualmente no GTKWave: acompanhar q_r3 (contador), q_r4 (acumulador) e q_r5 (resultado final).

4 · Extensão de Sinal — O Conceito Central do LAB6

O campo cte5 (bits [4:0] da instrução) é um número de 5 bits em complemento de 2. Para usar esse valor em operações de 16 bits, é necessário estender o sinal: replicar o bit mais significativo (bit 4) para preencher os 11 bits superiores.

Faixa representável em 5 bits com sinal

Binário (5 bits)Valor decimal
01111+15 (máximo positivo)
00001+1
000000
11111−1
11110−2
10000−16 (mínimo negativo)

Constantes acima de +15 (como 16, 30, 31) não cabem em 5 bits com sinal. Tentar codificá-las produz o valor negativo correspondente.

Como funciona a extensão de sinal

Se bit[4] = 0 (positivo): preenche com zeros.

00000000000
01111

→ 0x000F = +15 ✓

Se bit[4] = 1 (negativo): preenche com uns.

11111111111
11111

→ 0xFFFF = −1 ✓ (usado para incrementar R3)

Código VHDL da extensão de sinal (processador6.vhd:133)

-- Bug 1 (errado): zero-extension sempre — ignora o sinal do bit[4]
cte_ext_s <= "00000000000" & ir_out_s(4 downto 0);

-- Correto: extensão de sinal condicional
cte_ext_s <= "11111111111" & ir_out_s(4 downto 0) when ir_out_s(4) = '1'
             else "00000000000" & ir_out_s(4 downto 0);

Condição do flag BLE

BLE = 1 quando o resultado da ULA é menor ou igual a zero (com sinal):
BLE = Z OR (N XOR V)
onde Z=zero, N=negativo, V=overflow. Isso detecta corretamente "≤ 0" em complemento de 2. Usado para verificar se R3 < 30: calcula SUBI R0, R3, 29 (em dois passos de 15+14) → se R0 ≤ 0, BLE dispara e o loop continua.

6 · Programa de Teste na ROM

O programa calcula Σ i para i = 0, 1, 2, …, 29 usando R3 como contador e R4 como acumulador. O loop executa enquanto R3 < 30 (condição do enunciado). Ao final, R5 recebe uma cópia de R4.

  • A
    end 0: LD R3, 0 — inicializa contador R3 = 0.
  • B
    end 1: LD R4, 0 — inicializa acumulador R4 = 0.
  • C
    end 2: ADD R4, R3, R4 — R4 = R4 + R3 (acumula o valor atual do contador). Alvo do loop.
  • D
    end 3: SUBI R3, R3, −1 — R3 = R3 + 1 (incrementa; −(−1) = +1). Constante 11111 = −1 com sinal.
  • E
    end 4: SUBI R0, R3, 15 — R0 = R3 − 15 (passo 1 da comparação com 29).
  • F
    end 5: SUBI R0, R0, 14 — R0 = R0 − 14 = R3 − 29. Seta o flag BLE.
  • G
    end 6: BLE 2 — se R3 < 30 (R3−29 ≤ 0, BLE=1), volta para end 2 (ADD). Caso contrário segue em frente.
  • H
    end 7: MOV R5, R4 — copia R4 para R5. Executa apenas quando o loop termina.

Por que a comparação usa duas instruções SUBI?

O campo cte5 tem apenas 5 bits com sinal: range de −16 a +15. O valor 29 não cabe em uma única constante (máximo positivo é +15). A solução é subtrair em dois passos: 15 + 14 = 29:

SUBI R0, R3, 15 → R0 = R3 − 15
SUBI R0, R0, 14 → R0 = R3 − 29

Quando R3 ≤ 29 (i.e., R3 < 30): R0 ≤ 0 → BLE = 1 → loop continua.
Quando R3 = 30: R0 = 1 > 0 → BLE = 0 → loop encerra.

Verificação da codificação das instruções críticas

Como decodificar manualmente um binário de 15 bits

Formato: [14:11]=opcode, [10:8]=Rd, [7:5]=Rs1, [4:0]=cte5 (ou [4:2]=Rs2)

SUBI R3, R3, −1 = 001101101111111
opcode = 0011 (SUBI) | Rd = 011 (R3) | Rs1 = 011 (R3) | cte5 = 11111 (= −1) ✓

SUBI R0, R3, 15 = 001100001101111
opcode = 0011 (SUBI) | Rd = 000 (R0) | Rs1 = 011 (R3) | cte5 = 01111 (= +15) ✓

SUBI R0, R0, 15 = 001100000001111
opcode = 0011 (SUBI) | Rd = 000 (R0) | Rs1 = 000 (R0) | cte5 = 01111 (= +15) ✓

BLE 2 = 010100000000010
opcode = 0101 (BLE) | addr[6:0] = 0000010 (= 2 decimal) ✓

7 · Os 3 Bugs — Diagnóstico Completo

Três bugs distintos foram encontrados, cada um ocultando o seguinte. A correção precisou ser feita na ordem certa: Bug 1 → Bug 2 → Bug 3.

Bug 1 — Zero-extension ao invés de extensão de sinal (processador6.vhd:133)

Sintoma observado no GTKWave

R3 decrementava 31 unidades por iteração (ex: 0x0000, 0xFFE1, 0xFFC2…) em vez de incrementar 1. R4 acumulava valores negativos sem sentido.

Causa raiz

A instrução SUBI R3, R3, −1 codifica a constante −1 como 11111 em 5 bits. Com zero-extension, esse valor se torna 0x001F = 31 (não −1).

O SUBI então computa R3 − 31 em vez de R3 − (−1) = R3 + 1, causando o decremento de 31 visto no waveform.

Análise bit a bit

Valor5 bitsZero-ext (errado)Sign-ext (certo)
−1 11111 0x001F (+31) 0xFFFF (−1)
−2 11110 0x001E (+30) 0xFFFE (−2)
+15 01111 0x000F (+15) 0x000F (+15)

Correção — processador6.vhd:133

-- ANTES (bug): sempre prepend zeros → constante negativa vira positiva
cte_ext_s <= "00000000000" & ir_out_s(4 downto 0);

-- DEPOIS (correto): replica bit[4] nos 11 bits superiores
cte_ext_s <= "11111111111" & ir_out_s(4 downto 0) when ir_out_s(4) = '1'
             else "00000000000" & ir_out_s(4 downto 0);

Bug 2 — Constante 30 não representável em 5 bits com sinal (rom.vhd)

Sintoma após corrigir o Bug 1

R3 passou a incrementar corretamente (0, 1, 2, …), mas R4 ficava em 0 e o loop nunca terminava (loop infinito). A instrução de comparação estava calculando o valor errado.

A instrução original era SUBI R0, R3, 30 codificada como 001100001111110. O campo cte5 = 11110 que em 5 bits com sinal vale −2 (não +30).

Após o Bug 1 ser corrigido, o SUBI passou a calcular corretamente usando extensão de sinal: R0 = R3 − (−2) = R3 + 2. Para R3 ≥ 0, R0 sempre é positivo → BLE nunca dispara → loop infinito.

Por que 30 não cabe em 5 bits com sinal?

5 bits com sinal representam de −16 a +15. O valor 30 exigiria pelo menos 6 bits com sinal (range: −32 a +31). Qualquer constante acima de +15 é impossível de codificar diretamente. A solução é usar duas instruções somando constantes menores.

Correção — rom.vhd endereços 4–7

-- ANTES (bug): cte5=11110 = -2 com sinal → compara com -2, não 30
4 => "001100001111110",  -- SUBI R0, R3, 30  (BUG: cte5=11110=-2)
5 => "010100000000010",  -- BLE  2
6 => "010010110000000",  -- MOV  R5, R4

-- DEPOIS (correto): 29 = 15+14; compara R3 com 29 → BLE ≡ R3<30
4 => "001100001101111",  -- SUBI R0, R3, 15  (R0 = R3-15)
5 => "001100000001110",  -- SUBI R0, R0, 14  (R0 = R3-29)
6 => "010100000000010",  -- BLE  2
7 => "010010110000000",  -- MOV  R5, R4

Bug 3 — Flush dispara no estado errado do pipeline (uc6.vhd)

Sintoma após corrigir os Bugs 1 e 2

R3 incrementava corretamente, mas R4 ficava permanentemente em 0. A instrução ADD (end 2) nunca era executada — o MOV do fim do loop (end 7) executava no lugar da ADD a cada iteração.

Este é o bug mais sutil. Para entendê-lo é necessário traçar o pipeline ciclo a ciclo.

Como funciona o flush (conceito correto)

Quando um branch é tomado, a instrução seguinte ao branch já foi buscada da ROM (por causa da latência de 1 ciclo). Essa instrução não deve ser executada. O flush force ir_in = 0 (NOP) para descartar essa instrução espúria.

Chave: O IR só grava (wr_en_ir = 1) no estado 00. Portanto, o flush só tem efeito se estiver ativo durante o estado 00.

O Bug: flush ativado no estado 10 (wr_en_ir = 0 lá!)

-- CÓDIGO COM BUG: flush em estado=10, mas wr_en_ir=0 no estado 10!
flush_o <= '1' when (estado_s = "10" and is_jmp_s = '1') else
           '1' when (estado_s = "10" and branch_taken_s = '1') else
           '0';

Quando o estado muda para 00 (onde o IR de fato grava), o flush já estava em 0. O IR gravava a instrução espúria vinda da ROM.

Diagrama de temporização do Bug 3 (BLE em addr=6, target=2)

Execute BLE (estado=10) │ Fetch pós-BLE (00→01) │ Decode (01→10) ──────────────────────────────────────────┼───────────────────────┼─────────────────── PC (antes borda) 7 │ 1 (=target-1) │ 1 ROM captura addr 7 ← ainda é o addr 7! │ 1 (NOP area) │ 1 ROM saída (dado) conteudo_rom(7)=MOV │ NOP (correto) │ NOP ──────────────────────────────────────────┼───────────────────────┼─────────────────── flush (bug): 1 (INÚTIL! wr_en_ir=0) │ 0 ← flush sumiu! │ 0 wr_en_ir 0 │ 1 ← IR VAI GRAVAR │ 0 ir_in_s MOV (ignorado) │ MOV ← INSTRUÇÃO ERRADA│ ──────────────────────────────────────────┼───────────────────────┼─────────────────── IR (após borda) BLE (não mudou) │ MOV ← BUG! │ MOV Efeito: branch tomado OK │ NOP deveria ser gravado│ MOV executa!

O resultado: em cada iteração do loop, a instrução MOV R5, R4 executava (no lugar do NOP de flush), e a instrução ADD R4, R3, R4 era pulada completamente. Por isso R4 nunca acumulava nada.

Correção — uc6.vhd: flush no estado 00

Por que o estado=00 funciona?

Após o execute do BLE (estado 10 → 00), o IR ainda contém o opcode BLE (wr_en_ir foi 0 durante o execute). No estado 00, is_ble=1 e branch_taken=1 continuam válidos porque IR ainda não foi sobrescrito. Assim, ativar o flush em estado=00 funciona: wr_en_ir=1 e flush=1 → ir_in=0 → IR grava NOP ✓ 

-- DEPOIS (correto): flush em estado=00, onde wr_en_ir=1
flush_o <= '1' when (estado_s = "00" and is_jmp_s = '1') else
           '1' when (estado_s = "00" and branch_taken_s = '1') else
           '0';

Ajuste necessário no PC — target − 1

O flush insere um NOP que precisa passar pelos estados 01 e 10. No estado 01, o PC sempre incrementa +1. Se o branch saltasse diretamente para o endereço 2, depois do NOP o PC seria 3, pulando a instrução em 2.

A solução: armazenar o destino como target − 1. O NOP de bolha incrementa PC de target−1 para target. Resultado: a instrução correta é buscada. 

-- ANTES: PC vai direto para addr_abs_s
pc_next_o <= (pc_i + 1)        when estado_s = "01" else
             (pc_i + offset_s) when (estado_s = "10" and is_jmp_s = '1') else
             addr_abs_s        when (estado_s = "10" and branch_taken_s = '1') else
             pc_i;

-- DEPOIS: target-1 compensa o +1 do NOP de bolha no estado 01
pc_next_o <= (pc_i + 1)            when estado_s = "01" else
             (pc_i + offset_s - 1) when (estado_s = "10" and is_jmp_s = '1') else
             (addr_abs_s - 1)      when (estado_s = "10" and branch_taken_s = '1') else
             pc_i;

8 · Resumo dos 3 Bugs e Correções

  • 1
    Bug 1 — Zero-extension (processador6.vhd:133)
    Causa: cte_ext_s usava zero-extension; constante −1 (11111) tornava-se +31.
    Efeito: R3 decrementava 31 por iteração.
    Fix: Extensão de sinal condicional replicando bit[4].
  • 2
    Bug 2 — Constante inválida 30 em cte5 (rom.vhd)
    Causa: SUBI R0, R3, 30 codificava cte5=11110=−2; após Bug 1 fixado, comparava R3 com −2.
    Efeito: BLE nunca disparava → loop infinito.
    Fix: Duas instruções SUBI de +15 e +14 (total 29) para comparar R3<30; BLE e MOV deslocados em 1 endereço.
  • 3
    Bug 3 — Flush no estado errado (uc6.vhd)
    Causa: flush ativado em estado=10, mas wr_en_ir=0 lá; em estado=00 (onde IR grava), flush já era 0.
    Efeito: Instrução espúria (MOV) gravada no IR; ADD nunca executava; R4 ficava em 0.
    Fix: Mover flush para estado=00; ajustar PC de branch para target−1 para compensar bolha NOP.

9 · Verificação — O que Esperar no GTKWave

Após todas as correções, a simulação produz result_proc.ghw. No GTKWave, adicione os sinais nesta ordem:

rst, clk, estado_o, pc_o, instr_o, ula_out_o
e dentro do banco de registradores (via hierarquia):
q_r0, q_r3, q_r4, q_r5

Verificação da soma

Cálculo manual da soma esperada

Σ i para i = 0 a 29 = 29 × 30 / 2 = 435 = 0x01B3

Iteração por iteração (R3 no ADD → R4 após ADD):
i=0: R4 = 0+0 = 0 | i=1: R4 = 0+1 = 1 | i=2: R4 = 1+2 = 3 | i=3: R4 = 3+3 = 6 | …
i=28: R4 = 378+28 = 406 | i=29: R4 = 406+29 = 435 ✓
(quando R3 incrementa para 30, a condição R3<30 falha e o loop encerra)
